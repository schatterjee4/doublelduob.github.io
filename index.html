Part 3: 使用OAuth 2.0来让API更安全

Posted on 2015-09-19

原文链接

译文

在本篇博客中，我们使用OAuth2.0创建一个安全的API来让外部访问到我们在Part 1和Part 2中创建的微服务。

关于OAuth 2.0的更多资料，可以查阅资料Parecki - OAuth 2 Simplified和Jenkov - OAuth 2.0 Tutorial或者规范 IETF RFC 6749。

我们会添加一个新的product-api的微服务，它会作为对外的API（OAuth术语中的Resource Server）并且被Part 1中介绍的edge server暴露出去。我们还会添加一个OAuth Authorization Server和OAuth client。我们还会用到cURL。

Part 2中的系统架构图将会添加OAuth组件（红线标出的部分）：

我们会示范客户端如何使用四个标准权限认证流程中的一个，从authorization sever来获取访问token然后通过一个安全的请求访问resource server。

NOTES:

1.保护对外API并不是微服务特有的机制，所以本篇blog适用于任何需要使用OAuth2.0保护API的场景。

2.我们会提供一个轻量级的OAuth authorization server仅用来开发和测试。在生产环境下需要被API 平台或者第三方社交平台Facebook或者Twitter等替换掉。

3.我们在本篇博客中有意仅仅是用HTTP来简化复杂度。在任何真实场景下，所有OAuth的请求应该是用HTTPS。

4.我们强调过微服务和单体（monolithic）服务的区别在于每一个服务运行在一个独立的进程中。

AUTH-SERVER

authorization server的实现是很简单直接的。仅用了@EnableAuthorizationServer注解，然后通过配置类中指定的 client-id, client-secret, allowed grant flows和scopes属性，注册（in-memory）审批通过客户端应用。

@EnableAuthorizationServer
protected static class OAuth2Config extends AuthorizationServerConfigurerAdapter {

    @Override
    public void configure(ClientDetailsServiceConfigurer clients) throws Exception {
        clients.inMemory()
            .withClient("acme")
            .secret("acmesecret")
            .authorizedGrantTypes("authorization_code", "refresh_token", "implicit", "password", "client_credentials")
            .scopes("webshop");
    }
}

这个方式显然只在开发和测试场景下工作，目的是模拟一个real word的OAuth Authorization Server注册流程，如LinkedIn或Github。

完整代码请看AuthserverApplication.java

模拟Identity Provider（IdP）的用户注册信息，是通过在application.properties中添加一行user配置来实现的，如：

TRYING OUT THE FOUR OAUTH AUTHORIZATION GRANT FLOWS

OAuth 2.0规范定义了四种批准流程(grant flows)来获取access token：

详见Jenkov - OAuth 2.0 Authorization。

NOTE: 批准流程（grant flow）中Authorization Code和Implicit是最常用的，剩下两个可以被认为是前两个的边界情况（corner case）。
AUTHORIZATION CODE GRANT

首先我们需要通过浏览器获取一个code grant（它像一个一次性密码）：

http://localhost:9999/uaa/oauth/authorize?response_type=code&client_id=acme&redirect_uri=http://example.com&scope=webshop&state=97536

登录（user/password）然后在网页中批准。浏览器重定向的URL如：

http://example.com/?
code=IyJh4Y&
state=97536

NOTE: request中的state参数应该被设置为一个随机数，并且在response中被检查，来防止跨站请求伪造。

在重定向的URL中获取code参数，然后把它存在环境变量里：

CODE=IyJh4Y

现在作为安全服务器，用code来获取access token：

curl acme:acmesecret@localhost:9999/uaa/oauth/token \
 -d grant_type=authorization_code \
 -d client_id=acme \
 -d redirect_uri=http://example.com \
 -d code=$CODE -s | jq .
{
  "access_token": "eba6a974-3c33-48fb-9c2e-5978217ae727",
  "token_type": "bearer",
  "refresh_token": "0eebc878-145d-4df5-a1bc-69a7ef5a0bc3",
  "expires_in": 43105,
  "scope": "webshop"
}

把access token存入环境变量，后面访问API时使用：

TOKEN=eba6a974-3c33-48fb-9c2e-5978217ae727

让我们用同样的code在获取一次token，应该返回失败。code的真实作用是作为一次性密码：

curl acme:acmesecret@localhost:9999/uaa/oauth/token \
 -d grant_type=authorization_code \
 -d client_id=acme \
 -d redirect_uri=http://example.com \
 -d code=$CODE -s | jq .
{
  "error": "invalid_grant",
  "error_description": "Invalid authorization code: IyJh4Y"
}
IMPLICIT GRANT

在implicit grant的方式下，我们跳过了code的验证，从浏览器直接请求acceess token（降低了安全性）。在浏览器使用如下URL：

http://localhost:9999/uaa/oauth/authorize?
response_type=token&
client_id=acme&
redirect_uri=http://example.com&
scope=webshop&
state=48532

登录（user/password）然后在网页中批准。浏览器会重定向到类似如下URL：

http://example.com/#
  access_token=00d182dc-9f41-41cd-b37e-59de8f882703&
  token_type=bearer&
  state=48532&
  expires_in=42704

NOTE: request中的state参数应该被设置为一个随机数，并且在response中被检查，来防止跨站请求伪造。

把access token存入环境变量，后面访问API时使用：

TOKEN=00d182dc-9f41-41cd-b37e-59de8f882703
RESOURCE OWNER PASSWORD CREDENTIALS GRANT

这种场景下用户通常不能访问浏览器，而是要把自己的密码交给客户端程序，来获取一个access token（从安全的角度看，如果你不能信任客户端程序，这不是一个很好的解决方案）：

curl -s acme:acmesecret@localhost:9999/uaa/oauth/token  \
 -d grant_type=password \
 -d client_id=acme \
 -d scope=webshop \
 -d username=user \
 -d password=password | jq .
{
  "access_token": "62ca1eb0-b2a1-4f66-bcf4-2c0171bbb593",
  "token_type": "bearer",
  "refresh_token": "920fd8e6-1407-41cd-87ad-e7a07bd6337a",
  "expires_in": 43173,
  "scope": "webshop"
}

把access token存入环境变量，后面访问API时使用：

TOKEN=62ca1eb0-b2a1-4f66-bcf4-2c0171bbb593
CLIENT CREDENTIALS GRANT

在最后一种情况里，我们假设不需要用户授权来访问API（通常是一些不设计隐私的API）。客户端程序在这种情况下可以自己向鉴权服务器获取访问token：

curl -s acme:acmesecret@localhost:9999/uaa/oauth/token  \
 -d grant_type=client_credentials \
 -d scope=webshop | jq .
{
  "access_token": "8265eee1-1309-4481-a734-24a2a4f19299",
  "token_type": "bearer",
  "expires_in": 43189,
  "scope": "webshop"
}

把access token存入环境变量，后面访问API时使用：

TOKEN=8265eee1-1309-4481-a734-24a2a4f19299

请注意在这种情况下，access token代表的是客户端程序，而不是resource owner（用户），导致的问题是当product-api-service中 currentUser.getName()返回的将是acme而不是调用的用户。

ACCESSING THE API

当我们得到access token以后，我们就可以开始访问真正的API了。

首先尝试不用access token访问API，应该会失败：

curl 'http://localhost:8765/api/product/123' -s | jq .
{
  "error": "unauthorized",
  "error_description": "Full authentication is required to access this resource"
}

很好，是我们预期的结果。

然后，我们用一个不合法的token来访问，也应该会失败：

curl 'http://localhost:8765/api/product/123' \
 -H  "Authorization: Bearer invalid-access-token" -s | jq .
{
  "error": "access_denied",
  "error_description": "Unable to obtain a new access token for resource 'null'. The provider manager is not configured to support it."
}

我们又一次访问被拒绝。

现在，让我们用前面获得的合法的access token来请求：

curl 'http://localhost:8765/api/product/123' \
 -H  "Authorization: Bearer $TOKEN" -s | jq .
{
  "productId": 123,
  "name": "name",
  "weight": 123,
  "recommendations": [...],
  "reviews": [... ]
}

很好，我们得到了预期的结果。

来看api-service和product-api-service的日志打印：

2015-04-23 18:39:59.014  INFO 79321 --- [ XNIO-2 task-20] o.s.c.s.o.r.UserInfoTokenServices        : Getting user info from: http://localhost:9999/uaa/user
2015-04-23 18:39:59.030  INFO 79321 --- [ctApiService-10] s.c.m.a.p.service.ProductApiService      : ProductApi: User=user, Auth=Bearer a0f91d9e-00a6-4b61-a59f-9a084936e474, called with productId=123
2015-04-23 18:39:59.381  INFO 79321 --- [ctApiService-10] s.c.m.a.p.service.ProductApiService

我们可以发现，API向Authorization Server发送请求来获得用户的基本信息，然后打印在log中。

最终，我们来尝试如何让token失效，如模拟token过期。一个可行的方式是重启auth-server（我们的demo仅把token信息存在内存中）然后重新请求：

curl 'http://localhost:8765/api/product/123' \
 -H  "Authorization: Bearer $TOKEN" -s | jq .
{
  "error": "access_denied",
  "error_description": "Unable to obtain a new access token for resource 'null'. The provider manager is not configured to support it."
}

之前接受的access token如我们期望的，被拒绝了。

SUMMARY

开源项目spring-cloud-security和spring-security-auth让我们很容易的用OAuth 2.0搭建安全的API。不过要注意，例子中的Authorization Server智能用在开发和测试场景下。

Part 2: 尝试Eureka Hystrix（Circuit Breaker）防止微服务链式失败

Posted on 2015-08-09

原文链接

http://callistaenterprise.se/blogg/teknik/2015/04/15/building-microservices-with-spring-cloud-and-netflix-oss-part-2/

译文

在Part 1中，我们使用了诸如 Eureka, Ribbon和Zuul等几个Spring Cloud和Spring Cloud的核心组件，用来实现我们的操作模型（operations model）使得独立部署的微服务之间可以互相通信。本篇博客中，我们会关注微服务中的异常处理，使用Hystrix（Netflix的circuit breaker）来提升系统的还原能力。

现在一些异常情况将会在我们Part 1中搭建的系统中出现。如一些复合服务依赖的核心服务会突然的不返回响应，如果不正确的处理，符合服务便会收到影响。

通常我们把这类问题称为链式失败（chain of failures），一个组件的异常会导致依赖它的其他组件一起发生异常。在基于微服务的系统中，这类问题需要额外注意。一个常见的解决方案是使用断路器模式（circuit breaker pattern），更多信息可以从Release It!这本书和Fowler - Circuit Breaker这篇博客中获得。一个断路器的状态转换如下：

(Source: Release It!)

SPRING CLOUD AND NETFLIX OSS

从Part 1中提到的表格中我们会用到：Hystrix，Hystrix dashboard和Turbine。

  • Netflix Hystrix - Circuit breaker
    Netflix Hystrix给服务消费者提供了断路器保护的能力。如果一个服务不响应（如超时或者通信异常导致），Hystrix可以将调用重定向到服务消费者内部的fallback方法。如果一个服务重复的响应失败，Hystrix将会打开断路功能，在后续的调用中，直接调用内部的fallback方法，在服务恢复之前不去尝试调用服务。Hystrix会允许一部分请求去尝试服务是否返回，来判断服务是否可用。Hystrix嵌入服务消费者中运行。

  • Netflix Hystrix dashboard and Netflix Turbine - Monitor Dashboard
    Hystrix dashboard可以提供一个断路器的图形化的总览试图，Turbine（在基于Eureka的功能上）可以给dashboard提供所有断路器在系统中的信息。如下是一个Hystrix dashboard和Turbine的样例：

BUSINESS SERVICES

要启动Hystrix，需要在Spring Boot应用上使用@EnableCircuitBreaker注解。要真正使用Hystrix，我们还需要在想要监控的方法上添加@HystrixCommand的注解，并且设置一个fallback-method，如：

@HystrixCommand(fallbackMethod = "defaultReviews")
public ResponseEntity<List<Review>> getReviews(int productId) {
    ...
}

public ResponseEntity<List<Review>> defaultReviews(int productId) {
    ...
}

fallback方法被Hystrix用于异常情况（调用服务的方法失败或者超时等）或者当断路器启动时候的fast fail情况。完整的例子请看ProductCompositeIntegration.java。

Part 1: 使用 Netflix Eureka, Ribbon 和 Zuul构建微服务

Posted on 2015-07-29

原文链接

http://callistaenterprise.se/blogg/teknik/2015/04/10/building-microservices-with-spring-cloud-and-netflix-oss-part-1/

译文

在前面的博客中，我们定义了使用微服务时的operations model。在这篇博客中，我们来看一下如何用Spring Cloud和Netflix OSS来实现这个模型（model）。我们会涉及operations model中的几个核心部分：服务发现（service discovery），动态路由(dynamic routing)，负载均衡(load balancing)和边界服务器(edge server)，其他部分会在后续的博客中体现。

我们会使用Spring Cloud和Nexflix OSS中的一些核心组件，来让我们独立部署的微服务之间的交互时不用手工管理，比如管理每个微服务跑在哪个机器哪个端口，或者管理之间的路由关系。我们的微服务会在启动时设定一个动态分配的端口区间来防止端口冲突。我们还会用一个边界服务器作为入口，通过它来允许访问我们的微服务。

首先我们会简单的介绍Spring Could和Netflix OSS组件，之后我们将会给出一个这个系列博客都不断完善的系统结构。我们会从获取代码，编译构建开始，然后简单讲解代码中重要的部分，最终我们会简单的做一些测试来展示如何访问我们的微服务。启动新的微服务实例并让负载均衡组件工作都是非常容易的，不需要我们手工的配置。

SPRING CLOUD AND NETFLIX OSS

Spring Cloud是spring.io family中的新项目，它里面的一些组件会被我们用到。Spring Cloud 1.0 很大部分上基于Netflix OSS的开源组件。Spring Cloud用非常友好的自动配置和约定优于配置（convention over configuration）的方法集成了Netflix的组件到Spring环境中，这些友好的方式和Spring Boot的设计理念一致。

下面的表格是operations model和实际我们会使用的组件之间的对应关系：

我们会在这篇博客中使用Eureka，Ribbon和Zuul的一部分功能：

  • Netflix Eureka - 服务发现服务器（Service Discovery Server）
    Netflix Eureka使得微服务可以在运行时把自己注册到整个系统之中。

  • Netflix Ribbon - 动态路由和负载均衡(Dynamic Routing and Load Balancer)
    Netflix Ribbon 可以让服务消费者在运行时查询可用的服务。Ribbon通过Eureka提供的信息来找到合适的服务实例。如果大于一个实例被发现，Ribbon会应用负载均衡来使负载平均到每个实例上。Ribbon不是作为一个独立的服务，而是一个内嵌在服务消费者中的组件来提供服务。

  • Netflix Zuul - 边界服务器（Edge Server）
    Zuul是我们对外部世界的gatekeeper，禁止一切未被授权的外部访问通过。Zuul同样作为微服务对外部的入口。使用动态分配端口可以很方便的避免端口冲突的问题，也能减少管理成本，但是会给服务的使用增加一些困难。Zuul使用Ribbon来查询可用的服务，把外部的服务路由到内部合适的服务实例上。我们在这篇博客中会使用Zuul来提供一个入口（wellkown entrypoint），安全的问题会在后续的博客中解答。

注意: 那些通过边界服务器，允许外界访问的微服务就会被认为是系统层面的API。

A SYSTEM LANDSCAPE

我们需要一个系统结构来把各个组件串联起来。本篇博客中我们的会开发一个如下图所示的结构：

它包括四个业务服务（绿色部分）：

三个核心服务包括产品（products），推荐（recommendations）和审核（reviews）。
一个复合服务product-composite用来从其他三个核心服务组合（aggregate）信息，提供一个产品的信息并组合审核（reviews）和推荐的内容在其中。

我们使用了如下的基础服务和组件来支撑业务服务（蓝色部分）：

  • 服务发现服务器 Service Discovery Server (Netflix Eureka)

  • 动态路由和负载均衡 Dynamic Routing and Load Balancer (Netflix Ribbon)

  • 边界服务器 Edge Server (Netflix Zuul)

为了区别微服务和整块（monolithic）服务，我们会把每个服务分为单独的微服务。在一个很大规模的系统中这么细粒度的服务拆分显然是不合适的，取而代之的是把相关联的服务组织在一起，保证微服务的数量在一个适合管理的范围，但是也不会让服务过于板结在一起。

GRADLE DEPENDENCIES

和Spring Boot的设计思想相同，Spring Cloud定义了一些starter依赖，让我们可以非常容易的引入特定的功能特性。我们只要在build file里面添加如下内容，就可以在微服务中使用Eureka和Ribbon实现注册和调用其他微服务的功能：

完整的例子请看product-service/build.gradle。

compile('org.springframework.cloud:spring-cloud-starter-eureka-server:1.0.0.RELEASE')

Infrastructure servers

搭建基于Spring Cloud和Netflix OSS的基础服务器非常简单。比如搭建一个Eureka服务器仅需要在一个标准的Spring Boot应用中添加一个@EnableEurekaServer的注解：

@SpringBootApplication
@EnableEurekaServer
public class EurekaApplication {

public static void main(String[] args) {
    SpringApplication.run(EurekaApplication.class, args);
  }
}

完整的例子请看EurekaApplication.java。
搭建一个Zuul服务器添加 @EnableZuulProxy注解。完整的例子请看ZuulApplication.java。

仅用这些简单的注解就可以配置好服务器的默认配置并可以启动了。当我们需要特定的配置时可以覆盖默认的配置。一个例子是当我们需要限制什么服务可以被边界服务器允许被外部访问的时候，我们需要覆盖默认的配置做出修改。默认时Zuul设置的路由可以到Eureka中设置的每一个服务。我们用如下的配置即可限制路由只允许访问composite product这个服务的调用通过：

zuul:
ignoredServices: "*"
routes:
productcomposite:
  path: /productcomposite/**

完整的例子请看edge-server/application.yml
