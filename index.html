Part 2: 尝试Eureka Hystrix（Circuit Breaker）防止微服务链式失败

Posted on 2015-08-09

原文链接

http://callistaenterprise.se/blogg/teknik/2015/04/15/building-microservices-with-spring-cloud-and-netflix-oss-part-2/

译文

在Part 1中，我们使用了诸如 Eureka, Ribbon和Zuul等几个Spring Cloud和Spring Cloud的核心组件，用来实现我们的操作模型（operations model）使得独立部署的微服务之间可以互相通信。本片博客中，我们会关注微服务中的异常处理，使用Hystrix（Netflix的circuit breaker）来提升系统的还原能力。

现在一些异常情况将会在我们Part 1中搭建的系统中出现。如一些复合服务依赖的核心服务会突然的不返回响应，如果不正确的处理，符合服务便会收到影响。

通常我们把这类问题称为链式失败（chain of failures），一个组件的异常会导致依赖它的其他组件一起发生异常。在基于微服务的系统中，这类问题需要额外注意。一个常见的解决方案是使用断路器模式（circuit breaker pattern），更多信息可以从Release It!这本书和Fowler - Circuit Breaker这篇博客中获得。一个断路器的状态转换如下：

(Source: Release It!)

SPRING CLOUD AND NETFLIX OSS

从Part 1中提到的表格中我们会用到：Hystrix，Hystrix dashboard和Turbine。

  • Netflix Hystrix - Circuit breaker
    Netflix Hystrix给服务消费者提供了断路器保护的能力。如果一个服务不响应（如超时或者通信异常导致），Hystrix可以将调用重定向到服务消费者内部的fallback方法。如果一个服务重复的响应失败，Hystrix将会打开断路功能，在后续的调用中，直接调用内部的fallback方法，在服务恢复之前不去尝试调用服务。Hystrix会允许一部分请求去尝试服务是否返回，来判断服务是否可用。Hystrix嵌入服务消费者中运行。

  • Netflix Hystrix dashboard and Netflix Turbine - Monitor Dashboard
    Hystrix dashboard可以提供一个断路器的图形化的总览试图，Turbine（在基于Eureka的功能上）可以给dashboard提供所有断路器在系统中的信息。如下是一个Hystrix dashboard和Turbine的样例：

INFRASTRUCTURE SERVERS

搭建一个Turbine服务器需要添加一个 @EnableTurbineAmqp 的注解在Spring Boot的应用上:

@SpringBootApplication
@EnableTurbineAmqp
@EnableDiscoveryClient
public class TurbineApplication {

    public static void main(String[] args) {
        SpringApplication.run(TurbineApplication.class, args);
    }

}

完整例子请看TurbineApplication.java

搭建一个Hystrix Dashboard需要添加 @EnableHystrixDashboard注解。
这些注解就可以让我们用默认配置来开始使用了。当需要时，这些默认配置可以被指定的配置覆盖。

BUSINESS SERVICES

要启动Hystrix，需要在Spring Boot应用上使用@EnableCircuitBreaker注解。要真正使用Hystrix，我们还需要在想要监控的方法上添加@HystrixCommand的注解，并且设置一个fallback-method，如：

@HystrixCommand(fallbackMethod = "defaultReviews")
public ResponseEntity<List<Review>> getReviews(int productId) {
    ...
}

public ResponseEntity<List<Review>> defaultReviews(int productId) {
    ...
}

fallback方法被Hystrix用于异常情况（调用服务的方法失败或者超时等）或者当断路器启动时候的fast fail情况。完整的例子请看ProductCompositeIntegration.java。

Part 1: 使用 Netflix Eureka, Ribbon 和 Zuul构建微服务

Posted on 2015-07-29

原文链接

http://callistaenterprise.se/blogg/teknik/2015/04/10/building-microservices-with-spring-cloud-and-netflix-oss-part-1/

译文

在前面的博客中，我们定义了使用微服务时的operations model。在这篇博客中，我们来看一下如何用Spring Cloud和Netflix OSS来实现这个模型（model）。我们会涉及operations model中的几个核心部分：服务发现（service discovery），动态路由(dynamic routing)，负载均衡(load balancing)和边界服务器(edge server)，其他部分会在后续的博客中体现。

我们会使用Spring Cloud和Nexflix OSS中的一些核心组件，来让我们独立部署的微服务之间的交互时不用手工管理，比如管理每个微服务跑在哪个机器哪个端口，或者管理之间的路由关系。我们的微服务会在启动时设定一个动态分配的端口区间来防止端口冲突。我们还会用一个边界服务器作为入口，通过它来允许访问我们的微服务。

首先我们会简单的介绍Spring Could和Netflix OSS组件，之后我们将会给出一个这个系列博客都不断完善的系统结构。我们会从获取代码，编译构建开始，然后简单讲解代码中重要的部分，最终我们会简单的做一些测试来展示如何访问我们的微服务。启动新的微服务实例并让负载均衡组件工作都是非常容易的，不需要我们手工的配置。

SPRING CLOUD AND NETFLIX OSS

Spring Cloud是spring.io family中的新项目，它里面的一些组件会被我们用到。Spring Cloud 1.0 很大部分上基于Netflix OSS的开源组件。Spring Cloud用非常友好的自动配置和约定优于配置（convention over configuration）的方法集成了Netflix的组件到Spring环境中，这些友好的方式和Spring Boot的设计理念一致。

下面的表格是operations model和实际我们会使用的组件之间的对应关系：

我们会在这篇博客中使用Eureka，Ribbon和Zuul的一部分功能：

  • Netflix Eureka - 服务发现服务器（Service Discovery Server）
    Netflix Eureka使得微服务可以在运行时把自己注册到整个系统之中。

  • Netflix Ribbon - 动态路由和负载均衡(Dynamic Routing and Load Balancer)
    Netflix Ribbon 可以让服务消费者在运行时查询可用的服务。Ribbon通过Eureka提供的信息来找到合适的服务实例。如果大于一个实例被发现，Ribbon会应用负载均衡来使负载平均到每个实例上。Ribbon不是作为一个独立的服务，而是一个内嵌在服务消费者中的组件来提供服务。

  • Netflix Zuul - 边界服务器（Edge Server）
    Zuul是我们对外部世界的gatekeeper，禁止一切未被授权的外部访问通过。Zuul同样作为微服务对外部的入口。使用动态分配端口可以很方便的避免端口冲突的问题，也能减少管理成本，但是会给服务的使用增加一些困难。Zuul使用Ribbon来查询可用的服务，把外部的服务路由到内部合适的服务实例上。我们在这篇博客中会使用Zuul来提供一个入口（wellkown entrypoint），安全的问题会在后续的博客中解答。

注意: 那些通过边界服务器，允许外界访问的微服务就会被认为是系统层面的API。

A SYSTEM LANDSCAPE

我们需要一个系统结构来把各个组件串联起来。本篇博客中我们的会开发一个如下图所示的结构：

它包括四个业务服务（绿色部分）：

三个核心服务包括产品（products），推荐（recommendations）和审核（reviews）。
一个复合服务product-composite用来从其他三个核心服务组合（aggregate）信息，提供一个产品的信息并组合审核（reviews）和推荐的内容在其中。

我们使用了如下的基础服务和组件来支撑业务服务（蓝色部分）：

  • 服务发现服务器 Service Discovery Server (Netflix Eureka)

  • 动态路由和负载均衡 Dynamic Routing and Load Balancer (Netflix Ribbon)

  • 边界服务器 Edge Server (Netflix Zuul)

为了区别微服务和整块（monolithic）服务，我们会把每个服务分为单独的微服务。在一个很大规模的系统中这么细粒度的服务拆分显然是不合适的，取而代之的是把相关联的服务组织在一起，保证微服务的数量在一个适合管理的范围，但是也不会让服务过于板结在一起。

GRADLE DEPENDENCIES

和Spring Boot的设计思想相同，Spring Cloud定义了一些starter依赖，让我们可以非常容易的引入特定的功能特性。我们只要在build file里面添加如下内容，就可以在微服务中使用Eureka和Ribbon实现注册和调用其他微服务的功能：

完整的例子请看product-service/build.gradle。

完整例子请看discovery-server/build.gradle。

Infrastructure servers

搭建基于Spring Cloud和Netflix OSS的基础服务器非常简单。比如搭建一个Eureka服务器仅需要在一个标准的Spring Boot应用中添加一个@EnableEurekaServer的注解：

@SpringBootApplication
@EnableEurekaServer
public class EurekaApplication {

public static void main(String[] args) {
    SpringApplication.run(EurekaApplication.class, args);
  }
}

完整的例子请看EurekaApplication.java。
搭建一个Zuul服务器添加 @EnableZuulProxy注解。完整的例子请看ZuulApplication.java。

仅用这些简单的注解就可以配置好服务器的默认配置并可以启动了。当我们需要特定的配置时可以覆盖默认的配置。一个例子是当我们需要限制什么服务可以被边界服务器允许被外部访问的时候，我们需要覆盖默认的配置做出修改。默认时Zuul设置的路由可以到Eureka中设置的每一个服务。我们用如下的配置即可限制路由只允许访问composite product这个服务的调用通过：

zuul:
ignoredServices: "*"
routes:
productcomposite:
  path: /productcomposite/**

完整的例子请看edge-server/application.yml

这篇博客不是微服务的介绍，这有一个很好的介绍Fowler - Microservices给有需要的读者。博客中假定我们已经开始使用微服务来分解整块庞大（monolithic）的应用服务，用来提升可部署性（deployability）和扩展性（scalability）。

一个分布式系统中部署的微服务数量不断增加时与整块庞大（monothilic）服务的数量不断增加相比，新挑战的出现不会非常明显（译者注：因为微服务架构设计时照顾了水平扩展的需求）。这篇博客将会聚焦在这些大量微服务情况下出现的新挑战上，并会给大量微服务定义一个operations model。

Prerequisites先决条件

首先，我们来想一个问题，想要系统地运行（roll out）大量的微服务我们都需要些什么？
（原文：what is required to roll out large numbers of microservices in the system landscape?）

  • Composite services 组合服务

  • API services API服务

Scaling up

那么现在我们来看scale在这篇博客中会关注的地方：

当我们开始把整个庞大（monolithic）的应用服务拆分并替换为大量的微服务，会发现什么特点？

  • 1.大量的部署单元（deployed units）
    许多小的微服务相比大的完整的服务会导致大量的部署单元，需要我们去管理和监控

  • 2.各个微服务既是服务提供方也是服务调用方
    这会导致系统中的各个微服务将会互相交互

  • 3.一些服务对外暴露API
    这些微服务将会负责把其他微服务包裹起来，作为对外访问的接口

  • 4.整个系统将会更加的灵活（dynamic）
    新的微服务被部署，老的服务被替换或删除，系统中负载增大时我们可以增加微服务中对应的实例来解决问题。这意味着与之前相比服务会以更高的频率启停。

  • 5.比如平均故障间隔（MTBF）将会被降低，故障在整个系统的角度将会更频繁的发生。
    软件组件（components）时而会失败。随着大量小的部署单元的变多，整个系统中部分（或者非常小的部分）会失败的可能性相比整个庞大的服务应用相比会增加。

Questions

这将会导致一些重要的问题，有些情况会关系到运行时的问题：

  • 所有的微服务是如何配置的，它们配置的正确吗？
    当应用数量不多的时候管理配置并不是一个很难的问题。每个应用把自己的配置存在property文件中放在磁盘上，或者放在数据库的配置表中。当大量的微服务被部署到多个实例且运行在多个服务器的时候，这种方式变的非常难以控制。这会导致很多小的配置遍布到系统的各个部分，让整个系统想要高效高质量的维护变的很难。

  • 什么样的微服务需要被部署，部署在哪里？
    追踪服务在哪台机器哪个端口对外开放在数量不多的时候是一个成本很低的操作。当大量的微服务独立部署之后，并且每个微服务本身还会持续变更，这时如果还用人工维护的话，那会是一个噩梦。

  • 如何维护最新的路由信息？
    在一个动态变化的系统中作服务的调用方也会面对很多挑战。特别是当我们需要手工更新路由表，并且路由表在反向代理中或者在调用者的配置文件中时。根本上说，当我们在一个持续演化的系统中，系统的新的微服务随时会在新的机器或端口创建的时候，根本没有时间来让我们手动操作路由表。手工维护的交付时间相比之下就太慢了，因为变化次数的上升，手工操作出错的风险也会相比自动化配置有不必要的提高。

  • 如何防止链式失败？（chain of failures）
    当微服务之间相互调用时，我们就需要对链式失败进行额外的关注。如果一个被很多服务依赖的底层服务出问题了，会导致上层服务也出问题。如果不及时处理，大量服务会被一个出问题的服务拖累，导致我们的系统过于脆弱。

  • 如何保证所有服务都启动并且正常运行？
    追踪不多的应用很容易，我们如何保证大量的微服务时，服务的健康性呢？

  • 如何追踪在系统之间流转的信息？
    如果运营部门开始收到一些用户的抱怨，说我们的一些流程开始不正常了。
    哪个微服务才是错误的根源？
    我们如何能够找到那个出问题的流程？比如说订单号为12345的流程卡住，原因是微服务A不能访问，或者手动审批需要在微服务B之前才能发送确认消息。

  • 如何才能确保只有API-services暴露给外部？
    比如我们如何避免外界未授权的请求访问我们的内部？

  • 如何保证API－services的安全？
    这并不是关于微服务一个新的或者是特别的问题，但是当对外时，安全却是一个非常重要的问题。

Required components 必要组件

来给上述问题寻找答案，我们需要很多系统级别的新操作和管理的功能，但是当只操作少数应用服务时，这些功能其实是不太需要的。针对上述问题推荐的解决方案需要如下的组建：

  • Central Configuration server 中央配置服务器
    我们需要一个中央配置管理服务，而不是每个部署单元（如一个微服务）都靠一个本地的配置文件。我们同样需要配置的API来让微服务获取配置信息。

  • Service Discovery server 服务发现服务器
    我们需要一个服务发现的功能来让我们通过API使得微服务可以自主的在启动时注册服务，而不是手工的管理追踪服务被部署在哪个机器的哪个端口。

  • Dynamic Routing and Load Balancer 动态路由和负载均衡
    路由组件在配合服务发现功能后可以使用服务发现API来查询需要的服务被部署在哪里，并且负载均衡组件可以决定多个实例中的哪一个实例来响应请求。

  • Circuit Breaker 断路器
    To avoid the chain of failures problem we need to apply the Circuit Breaker pattern, for details see the book Release It! or read the blog post Fowler - Circuit Breaker.
    来避免链式失败问题，我们需要应用断路器模式，预知详情，请看Release It! 或者看这篇博客Fowler - Circuit Breaker。

  • Monitoring 监控
    在断路器的配合下，我们可以监控断路器的状态，同时收集运行时信息来获得一个系统级别的健康的图谱。
    这些信息可以在dashboard上收集和展示，并设置阈值自动报警。

  • Centralized log analysis 集中式的日志处理
    当服务卡住的时候，为了有能力追踪和发现问题，我们需要一个集中式的日志处理功能，它能访问到服务器并且收集每个微服务产生的日志文件。日志收集功能把把日志储存在一个中央日志系统，来提供搜索和展示的能力。
    注意：为了有能力找到相护关联的信息，所有微服务之间在log中使用一个相关的id是非常重要的。

  • Edge Server 边界服务器
    为了给外部暴露API服务并且防止未授权的请求访问内部服务，我们需要一个边界服务器，所有的外部访问都经过它们进入系统。边界服务器可以在上述服务发现组件的功能上复用动态路由和负载均衡的能力。边界服务器将会扮演一个动态和主动的反向代理，并且在内部系统发生变化的时候不需要手工更新。

  • OAuth 2.0 protected API’s 在OAuth 2.0保护下的API
    对于防护对外的API服务，推荐使用OAuth 2.0标准。
    使用OAuth 2.0的我们需要：

    • 一个新的组件作为OAuth鉴权服务器
    • API服务会作为OAuth的资源服务器（OAuth Resource Server）
    • 外部的API调用作为OAuth的客户（OAuth Clients）
    • 边界服务器会作为OAuth的Token Relay意味着：
      • 它会作为OAuth的资源服务器（OAuth Resource Server）
      • 它会让OAuth访问从外部通过，然后访问内部API服务。

    注意：随着时间的推移，OAuth 2.0标准很有可能会被OpenID Connect标准完全替代，来提供更好的鉴权功能。

  • 点击左下角的文件，review代码，然后根据自己的意见打分（为什么要codereview），如果一切顺利，admin或者owner觉得没问题，会看到下面的样子, 绿色字样Core-Review +2表示评审通过，等待admin或者owner点击submit合并到对应分支，这时候就review的提交者就可以去做别的事了，

  • 当然，拿出来review总是要还的，如果－2分不通过，则需要看看评审人的意见，然后做出修改
    修改完成后，git add 修改的文件, 然后执行git commit —amend，说明作出的调整，然后重新git push origin HEAD:refs/for/master发出评审

  • 可以看到右上角的Patch Sets变成了2，点击以后可以选择，评审页面也可以选择，review后通过后submit就可以merge代码，打回后继续修改

  • 如果你觉得git push origin HEAD:refs/for/master这个命令很麻烦，可以了解一下git-review这个命令行小工具

  • 如果你不喜欢命令行，喜欢用intellij idea或者eclipse或者或者其他GUI的工具，git gerrit也是有相关工具支持的，可以google一下

GIT和SVN特点的详细比较 介绍了GIT对比SVN的一些优点

面向 Subversion 用户的 Git，入门指南 先从SVN的常见操作入手，再用GIT复现一遍

如果你英文不错，可以看看SVN_CRASH_COURSE

当然如果你想直接从头学习GIT，这里有个猴子都能懂的GIT入门

常用命令对照

操作 Git Subversion
迁出 git clone svn checkout
提交 git commit svn commit
查看提交的详细记录 git show svn cat
确认状态 git status svn status
确认差异 git diff svn diff
确认记录 git log svn log
添加 git add svn add
移动 git mv svn mv
删除 git rm svn rm
取消修改 git checkout / git reset svn revert (※1)
创建分支 git branch svn copy (※2)
切换分支 git checkout svn switch
合并 git merge svn merge
创建标签 git tag svn copy (※2)
更新 git pull / git fetch svn update
反映到远端 git push svn commit (※3)
忽略档案目录 .gitignore .svnignore 
※1. SVN的revert是用来取消修改，但Git的revert是用来消除提交。所以即使是同样的命令，在SVN和Git里的含义是不同的。

※2. SVN的分支与标签在构造上是相同的，但在Git其构造明显是不一样的。

※3. SVN没有本地数据库/远程数据库的概念，所以提交会马上反映到远程里。但Git的本地数据库和远程数据库的反映方法是不一样的。
