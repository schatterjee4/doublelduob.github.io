Part 2: 尝试Eureka Hystrix（Circuit Breaker）防止微服务链式失败

Posted on 2015-08-09

原文链接

http://callistaenterprise.se/blogg/teknik/2015/04/15/building-microservices-with-spring-cloud-and-netflix-oss-part-2/

译文

在Part 1中，我们使用了诸如 Eureka, Ribbon和Zuul等几个Spring Cloud和Spring Cloud的核心组件，用来实现我们的操作模型（operations model）使得独立部署的微服务之间可以互相通信。本篇博客中，我们会关注微服务中的异常处理，使用Hystrix（Netflix的circuit breaker）来提升系统的还原能力。

现在一些异常情况将会在我们Part 1中搭建的系统中出现。如一些复合服务依赖的核心服务会突然的不返回响应，如果不正确的处理，符合服务便会收到影响。

通常我们把这类问题称为链式失败（chain of failures），一个组件的异常会导致依赖它的其他组件一起发生异常。在基于微服务的系统中，这类问题需要额外注意。一个常见的解决方案是使用断路器模式（circuit breaker pattern），更多信息可以从Release It!这本书和Fowler - Circuit Breaker这篇博客中获得。一个断路器的状态转换如下：

(Source: Release It!)

SPRING CLOUD AND NETFLIX OSS

从Part 1中提到的表格中我们会用到：Hystrix，Hystrix dashboard和Turbine。

  • Netflix Hystrix - Circuit breaker
    Netflix Hystrix给服务消费者提供了断路器保护的能力。如果一个服务不响应（如超时或者通信异常导致），Hystrix可以将调用重定向到服务消费者内部的fallback方法。如果一个服务重复的响应失败，Hystrix将会打开断路功能，在后续的调用中，直接调用内部的fallback方法，在服务恢复之前不去尝试调用服务。Hystrix会允许一部分请求去尝试服务是否返回，来判断服务是否可用。Hystrix嵌入服务消费者中运行。

  • Netflix Hystrix dashboard and Netflix Turbine - Monitor Dashboard
    Hystrix dashboard可以提供一个断路器的图形化的总览试图，Turbine（在基于Eureka的功能上）可以给dashboard提供所有断路器在系统中的信息。如下是一个Hystrix dashboard和Turbine的样例：

SOURCE CODE WALKTHROUGH

相比Part 1新增的部分就是Hystrix在product-composite服务中作为断路器，我们会聚焦在新增的代码上。

GRADLE DEPENDENCIES

我们有一些Hystrix-based starter依赖可以直接放到build file中。因为Hysteric使用RabbitMQ来让断路器和dashboard通信，我们还需额外配置一下。

对于服务的消费者，想使用Hystrix作为断路器，我们需要添加：

compile("org.springframework.cloud:spring-cloud-starter-hystrix:1.0.0.RELEASE")
compile("org.springframework.cloud:spring-cloud-starter-bus-amqp:1.0.0.RELEASE")
compile("org.springframework.cloud:spring-cloud-netflix-hystrix-amqp:1.0.0.RELEASE")

完整的例子请看product-composite-service/build.gradle.
要想启动Turbine的服务器，需要添加如下依赖：

compile('org.springframework.cloud:spring-cloud-starter-turbine-amqp:1.0.0.RELEASE')

完整内容请看turbine/build.gradle.

INFRASTRUCTURE SERVERS

搭建一个Turbine服务器需要添加一个 @EnableTurbineAmqp 的注解在Spring Boot的应用上:

@SpringBootApplication
@EnableTurbineAmqp
@EnableDiscoveryClient
public class TurbineApplication {

    public static void main(String[] args) {
        SpringApplication.run(TurbineApplication.class, args);
    }

}

完整例子请看TurbineApplication.java

搭建一个Hystrix Dashboard需要添加 @EnableHystrixDashboard注解。
这些注解就可以让我们用默认配置来开始使用了。当需要时，这些默认配置可以被指定的配置覆盖。

BUSINESS SERVICES

要启动Hystrix，需要在Spring Boot应用上使用@EnableCircuitBreaker注解。要真正使用Hystrix，我们还需要在想要监控的方法上添加@HystrixCommand的注解，并且设置一个fallback-method，如：

@HystrixCommand(fallbackMethod = "defaultReviews")
public ResponseEntity<List<Review>> getReviews(int productId) {
    ...
}

public ResponseEntity<List<Review>> defaultReviews(int productId) {
    ...
}

fallback方法被Hystrix用于异常情况（调用服务的方法失败或者超时等）或者当断路器启动时候的fast fail情况。完整的例子请看ProductCompositeIntegration.java。

正如Part 1中提到的，我们要在本地环境启动微服务的java进程。我们需要安装cURL和jq工具。

如我们提过的，Hystrix使用RabbitMQ来进行内部通讯，所以我们需要在系统启动前安装并运行它。执行如下命令进行下载和安装，使用sbin目录下的rabbitmq－server程序来启动RabbitMQ：

$ cd core/product-service;                ./gradlew bootRun
$ cd core/recommendation-service;         ./gradlew bootRun
$ cd core/review-service;                 ./gradlew bootRun

SOMETHING GOES WRONG

停掉 review 微服务然后重试如下命令：

$ curl -s localhost:8765/productcomposite/product/1 | jq .
{
    "name": "name",
    "productId": 1,
    "recommendations": [
    {
        "author": "Author 1",
        "rate": 1,
        "recommendationId": 0
    },
    ...
    ],
    "reviews": null,
    "weight": 123
}

review部分的响应（response）现在是空的了，但是其他部分仍然是完整的。我们来看下product-composite服务的日志，你会看到如下警告：

2015-04-02 15:13:36.344  INFO 29901 --- [eIntegration-10] s.c.m.c.p.s.ProductCompositeIntegration  : GetRecommendations...
2015-04-02 15:13:36.497  INFO 29901 --- [teIntegration-2] s.c.m.c.p.s.ProductCompositeIntegration  : GetReviews...
2015-04-02 15:13:36.498  WARN 29901 --- [teIntegration-2] s.c.m.composite.product.service.Util     : Failed to resolve serviceId 'review'. Fallback to URL 'http://localhost:8081/review'.
2015-04-02 15:13:36.500  WARN 29901 --- [teIntegration-2] s.c.m.c.p.s.ProductCompositeIntegration  : Using fallback method for review-service

circuit breaker发现了review服务的问题，把请求路由到服务调用者的fallback方法。在这个例子里，我们仅仅return null，但是我们也可以在review service不可用时返回一个缓存数据。

error的频率很低，所以circuit breaker没有开启：

我们来增加error的发生频率，Hystrix会打开circuit然后快速失败防止请求阻塞（我们使用Hystrix的默认值来让blog中的内容尽量简洁）。我们用Apache HTTP服务器的benchmarking工具来测试：

ab -n 30 -c 5 localhost:8765/productcomposite/product/1

现在断路器将会被激活：

SUMMARY

我们已经看到Netflix Hystrix可以被用做circuit breaker来高效的处理链式失败请求问题。Spring Cloud提供了非常方便的注解和starter依赖使得Hystrix可以很容易的集成到Spring的环境中。最后Hystrix dashboard和Turbine给了我们监控了整个系统的请求状态的能力。
