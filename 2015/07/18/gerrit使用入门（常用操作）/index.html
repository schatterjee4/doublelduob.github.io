gerrit使用入门（常用操作）

Posted on 2015-07-18

目的

在完成gerrit的基本配置之后，帮助用户熟悉gerrit的基本的评审流程，不会涉及^gerrit是什么，^为什么要codereview的部分，更详细的gerrit的使用，可以参考^Learning Gerrit Code Review，使用部分为chapter6，7

准备

已经有可用的gerrit服务，并且有权限clone上面的某个项目，并完成的基本配置

步骤

  • 此时你刚配置好gerrit，clone了代码，并开发了新的feature或改了一个bug，git add 修改的文件，git commit －m “描述这个feature或bug”(如果你不想每次都一个个选择修改的文件，可以把不想提交的文件放到gitignore中，然后git commit -a或git commit —all)

  • 运行git log，查看提交纪录，会看到里面有Change-Id这部分内容，这个Change-Id是gerrit用来标识一次修改的，如果没有看到，可以参考配置部分，拷贝那条clone命令的后半段，如scp -p -P 29418 admin@localhost:hooks/commit-msg demo/.git/hooks/在项目的上级目录(执行clone的目录)执行，失败说明SSH配置不成功，参考配置部分

  • 提交前先更新代码，建议运行git pull -r（如果你还不太了解git的分支，或者你的gerrit项目禁用了merge提交需要rebase代码，直接pull下来会形成本地merge），然后运行git push origin HEAD:refs/for/master，把修改提交到gerrit中repo的master分支上

  • 打开里面的链接，进入gerrit的web页面，如果你不是项目的owner或者admin，那么你看到的应该是下面的样子

  • 点开replay按钮的样子，＋1表示你认为代码变更没问题，－1表示有问题，然后加上自己的评论，0表示路过

  • 这时候需要添加评审人，可以是项目的owner和组内相同功能的backup小伙伴，他们中有权限的会看到下面的样子，＋2表示一定没问题，可以merge到master分支，－2表示一定有问题，需要修改，如果点击了＋2，页面会有submit按钮可点，点击后代码和并入对应分支（一般是master）

  • 点击左下角的文件，review代码，然后根据自己的意见打分（^为什么要codereview），如果一切顺利，admin或者owner觉得没问题，会看到下面的样子, 绿色字样Core-Review +2表示评审通过，等待admin或者owner点击submit合并到对应分支，这时候就review的提交者就可以去做别的事了，

  • 当然，拿出来review总是要还的，如果－2分不通过，则需要看看评审人的意见，然后做出修改
    修改完成后，git add 修改的文件, 然后执行git commit —amend，说明作出的调整，然后重新git push origin HEAD:refs/for/master发出评审

  • 可以看到右上角的Patch Sets变成了2，点击以后可以选择，评审页面也可以选择，review后通过后submit就可以merge代码，打回后继续修改

  • 如果你觉得git push origin HEAD:refs/for/master这个命令很麻烦，可以了解一下^git-review这个命令行小工具

  • 如果你不喜欢命令行，喜欢用intellij idea或者eclipse或者或者其他GUI的工具，git gerrit也是有相关工具支持的，可以google一下
